# GramGPT — Техническое задание на аудит безопасности аккаунтов

## Цель аудита

Провести полный аудит кодовой базы GramGPT на предмет безопасности Telegram-аккаунтов. Главная задача — найти и устранить все места в коде, которые могут привести к бану, заморозке или сбросу сессий аккаунтов Telegram.

## Критерии безопасности (приоритет от высокого к низкому)

1. **Все подключения к Telegram ТОЛЬКО через прокси** — ни один запрос не должен идти напрямую
2. **Минимум подключений Telethon** — одно подключение на аккаунт за сессию, не на каждое действие
3. **Человекоподобное поведение** — рандомные задержки, расписание сессий, дни отдыха
4. **Лимиты действий** — не превышать ограничения Telegram API
5. **Консистентный fingerprint** — один аккаунт = одно устройство навсегда
6. **Защита credentials** — никаких ключей в git, session файлы в безопасности

## Критические точки для аудита

### 1. Подключение к Telegram (`api/utils/telegram.py`)

**Функция `make_telethon_client(account, proxy)`** — единственная точка создания Telethon клиента. Все модули ОБЯЗАНЫ использовать её.

**Что проверить:**
- Если `proxy` = None, функция НЕ ДОЛЖНА создавать клиент (должна вернуть None)
- Device fingerprint должен быть детерминированным (один номер = одно устройство навсегда)
- Для существующих аккаунтов (session файл уже есть) НЕ менять device_model — Telegram убьёт сессии
- `api_id` / `api_hash` берутся из `account.api_app` если есть, иначе из глобального конфига
- Session path корректный и файл существует

**Возможные проблемы:**
- Подключение без прокси → Telegram видит реальный IP → бан
- Смена device_model на существующем аккаунте → сброс всех сессий
- Использование одного api_id на 50+ аккаунтов → массовый бан

### 2. Прогрев аккаунтов (`api/tasks/warmup_v2.py`)

**Модель: сессии живого человека**

Расписание (UTC+3):
- 8:00-10:00 — утренняя проверка (5-10 действий)
- 10:00-12:00 — мини-проверка (1-3 действия)
- 12:00-14:00 — обеденный залип (8-15 действий)
- 14:00-16:00 — мини-проверка (1-3 действия)
- 16:00-19:00 — мини-проверка (2-5 действий)
- 19:00-24:00 — вечерний сёрфинг (10-25 действий)

Типы дней:
- Normal (40%) — обычный день
- Active (20%) — больше действий (×1.3)
- Lazy (20%) — только 1-2 захода (×0.5)
- Random (15%) — хаотичные часы (×0.8)
- Rest (5%) — полный отдых, 0 действий

**Что проверить:**
- Каждый аккаунт подключается ОДИН РАЗ за сессию (connect → все действия → disconnect)
- Между действиями внутри сессии паузы 10-60с
- Между сессиями тишина 1-3 часа (не подключается к Telegram)
- Рандомный сдвиг времени начала сессии ±60-90 минут
- Градация по дням: день 1 = мало действий, день 7 = полная активность
- Часовой пояс: `hour = (now.hour + 3) % 24` для UTC+3
- `next_action_at` правильно сохраняется и проверяется
- Логи пишутся с `created_at=datetime.utcnow()` явно, а не default
- `db.flush()` + `db.commit()` после каждого лога (не в конце сессии)

**Действия прогрева (вес = частота):**
| Действие | Вес | Описание |
|----------|-----|----------|
| read_feed | 25 | Читает сообщения в случайном диалоге, помечает прочитанными |
| set_reaction | 18 | Ставит случайный эмодзи на пост в канале |
| view_stories | 12 | GetAllStoriesRequest |
| view_profile | 8 | GetFullUserRequest случайного контакта |
| typing | 5 | SetTypingAction в Saved Messages |
| search | 5 | SearchRequest случайного слова |
| join_channel | 4 | JoinChannel в популярный канал |
| forward_saved | 5 | Пересылка поста в Saved Messages |
| send_saved | 10 | Пишет случайное сообщение в Saved Messages |
| reply_dm | 8 | Отвечает на непрочитанные ЛС текстом или стикером |

**Возможные проблемы:**
- Слишком много join_channel за день → CHANNELS_TOO_MUCH
- Реакции на каналах где они отключены → REACTION_INVALID (нужно ловить)
- Typing без отправки сообщения слишком часто → подозрительно
- Один и тот же набор действий каждый день → паттерн

### 3. Комментинг (`api/tasks/commenting_tasks.py`)

**Процесс:**
1. `run_periodic.py` вызывает `process_campaigns` каждые 90с
2. Для каждой активной кампании парсит каналы через HTTPS (веб-парсинг, НЕ Telethon)
3. Если новый пост → проверяет триггер (all/random %/keywords)
4. Генерирует комментарий через LLM (Claude/GPT-4o/Gemini/Groq)
5. Подключает случайный аккаунт через Telethon + прокси
6. Отправляет `send_message(entity, comment, comment_to=post_id)`
7. Отключается

**Что проверить:**
- Веб-парсинг (`fetch_latest_posts_web`) не использует Telethon — OK
- Отправка комментария — одно короткое подключение на один коммент
- `delay_comment` (250с по умолчанию) — задержка перед комментарием после нового поста
- `delay_between` (60с) — между комментариями в разных каналах
- Случайный выбор аккаунта из `campaign.account_ids`
- Аккаунт без прокси → НЕ подключается (сейчас просто предупреждение в лог, нужно пропускать!)
- `delay_join` — сейчас НЕ используется в коде (поле есть, логика отсутствует)
- Нет проверки подписан ли аккаунт на канал перед комментированием

**Возможные проблемы:**
- Комментирование без подписки на канал → подозрительно для модераторов
- Один аккаунт комментирует слишком часто → нужен per-account лимит
- Подключение без прокси если прокси не назначен
- Нет rate limiting per account (один аккаунт может получить 10 комментариев подряд)

### 4. Предподписка (`api/tasks/subscribe_tasks.py`)

**Процесс:**
1. Пользователь выбирает кампанию → нажимает "Подписать"
2. Создаётся задача с account_ids + channels + total_minutes
3. Celery запускает в фоне
4. Для КАЖДОГО аккаунта — ОДНО подключение через прокси
5. Подписка на все каналы с рандомными паузами
6. Между аккаунтами — большие паузы

**Что проверить:**
- Одно подключение на аккаунт (connect → все JoinChannel → disconnect) — OK
- Аккаунт без прокси пропускается — OK
- Обработка FLOOD_WAIT (ждёт + рандомная добавка) — OK
- Обработка CHANNELS_TOO_MUCH (прерывает для этого аккаунта) — OK
- Паузы между каналами (15с–10мин) рассчитываются из total_minutes — OK
- Паузы между аккаунтами (30% от time_per_account) — OK

### 5. Реакции (`api/routers/reactions.py`)

**Процесс:**
1. Для каждого аккаунта — отдельное подключение
2. SendReactionRequest на пост и/или комментарии
3. Для комментариев — GetFullChannelRequest → linked_chat_id → iter_messages

**Что проверить:**
- Каждый аккаунт = отдельное подключение (НЕ одно на всех)
- Задержки между аккаунтами (delay_min – delay_max)
- Для target="comments" — подключение к discussion group через linked_chat_id
- Обработка REACTION_INVALID

**Возможные проблемы:**
- Если 50 аккаунтов ставят реакцию на один пост одновременно → подозрительно
- Нет проверки что аккаунт подписан на канал для реакции

### 6. Авторизация (`api/routers/tg_auth.py`)

**Процесс:**
1. send-code → создаёт Telethon клиент → client.send_code_request()
2. confirm → client.sign_in(code) → если 2FA → возвращает needs_2fa=true
3. confirm-2fa → client.sign_in(password)

**Что проверить:**
- Клиент создаётся через `_make_client()` — НЕ через `make_telethon_client()` (отдельная логика!)
- `_make_client()` использует ФИКСИРОВАННЫЙ device_model="Desktop" (не рандомный!)
- Прокси передаётся из PENDING_PROXY
- ACTIVE_CLIENTS хранит живые клиенты в памяти между send-code и confirm
- 2FA обработка: SessionPasswordNeededError → needs_2fa=true → frontend показывает поле пароля

**Возможные проблемы:**
- `_make_client()` использует `cli_config.API_ID` из `../../config.py` — НЕ из api_app аккаунта!
- Если перезапустить uvicorn между send-code и confirm → ACTIVE_CLIENTS потерян → ошибка
- `load_cli_config()` загружает `../../config.py` который конфликтует с `api/config.py`

### 7. Мульти-API система (`api/models/api_app.py`, `api/services/api_apps.py`)

**Зачем:** Распределить аккаунты по разным api_id/api_hash чтобы избежать массового бана.

**Что проверить:**
- Новый аккаунт при импорте получает api_app через `pick_best_app()` (наименее нагруженный)
- Аккаунт НАВСЕГДА привязан к одному api_app (менять нельзя — session ломается)
- `make_telethon_client()` берёт api_id/api_hash из `account.api_app` если есть

### 8. config.py конфликт

**Проблема:** Есть два config.py:
- `tg_manager/config.py` — для CLI инструментов (без DATABASE_URL)
- `tg_manager/api/config.py` — для API (с DATABASE_URL)

Celery worker при выполнении тасков иногда импортирует неправильный config.py.

**Текущий фикс:** `sys.path.insert(0, API_DIR)` в начале каждого таска.

**Что проверить:**
- Все таски имеют `API_DIR = ...` и `sys.path.insert(0, API_DIR)` в начале
- celery_app.py НЕ должен трогать sys.path (ломает остальные таски)
- `tg_auth.py` использует `load_cli_config()` для доступа к корневому config — это отдельная проблема

## Задача 7: Максимально человекоподобное поведение

Это ГЛАВНАЯ задача аудита. Все действия должны быть неотличимы от реального пользователя Telegram. Никаких паттернов, никакой предсказуемости.

### 7.1 Прогрев — требования к человечности

**Текущая модель:** сессии (утро/обед/вечер) с рандомными действиями и паузами.

**Что проверить и улучшить:**

- **Нет двух одинаковых дней** — даже одинаковый тип дня (normal) должен давать разный набор действий, в разном порядке, с разными паузами
- **Рандом в рандоме** — не просто "пауза 30-60с", а иногда 5с (быстро пролистал), иногда 3 минуты (залип на посте), иногда 10 минут (отвлёкся)
- **Контекстные действия** — после read_feed логично поставить реакцию на то что прочитал, а не сразу search. Добавить "цепочки действий": прочитал канал → реакция → пересылка в saved (как реальный человек сохраняет интересное)
- **Не все действия каждый день** — если сегодня "ленивый день", то только read_feed и view_stories. Никаких join_channel и forward_saved
- **Разные каналы** — не одни и те же 3 канала каждый день, а из всех подписок случайно
- **Реалистичное чтение** — не всегда 10 сообщений, иногда 1-2 (глянул и закрыл), иногда 20+ (залип)
- **Время между сессиями** — не фиксированное 1-3 часа, а зависит от типа дня. Ленивый день: 4-6 часов между заходами. Активный: 30-60 минут
- **Постепенное вовлечение** — день 1-2: только read_feed и view_stories. День 3-4: добавляются реакции. День 5+: полный набор действий с reply_dm и send_saved

### 7.2 Комментинг — требования к человечности

**Критически важно — комментинг это главный источник банов.**

**Что проверить и улучшить:**

- **Per-account лимит** — один аккаунт НЕ БОЛЕЕ 3-5 комментариев в день. Сейчас этого лимита НЕТ — один аккаунт может получить 20 комментариев подряд
- **Ротация аккаунтов** — не random.choice (может дать один аккаунт 5 раз подряд), а round-robin с рандомизацией
- **Пауза после комментария** — аккаунт после комментирования "отдыхает" минимум 2-4 часа перед следующим комментарием
- **Предварительное чтение** — перед комментарием аккаунт должен "прочитать" канал (send_read_acknowledge), подождать 30-120с, и только потом комментировать. Реальный человек не комментирует мгновенно
- **Не комментировать первым** — delay_comment минимум 300-600с (5-10 минут). Первый комментарий от нового аккаунта подозрителен
- **Разное время комментирования** — не все аккаунты комментируют с одинаковой задержкой. Один через 5 минут, другой через 40 минут, третий через 2 часа
- **Не комментировать каждый пост** — даже в режиме "all" реальный человек пропускает неинтересные посты. Добавить фильтр: пропускать посты короче 50 символов, посты-картинки без текста, рекламные посты
- **Поведение перед комментарием** — аккаунт должен быть "живым": утром зашёл, почитал, поставил пару реакций, и только потом в обед написал комментарий. Не "подключился → написал → отключился"
- **Качество комментариев** — AI должен генерировать разнообразные комментарии. Не шаблонные "Отличный пост!", а с контекстом, вопросами, личным мнением. Промпт для LLM должен включать: "пиши с ошибками иногда, используй разговорный стиль, иногда соглашайся не полностью"
- **Язык комментария** — автоматически определять язык поста и комментировать на том же языке

### 7.3 Предподписка — требования к человечности

- **Не подписываться на все каналы за один заход** — разбить на несколько дней. День 1: 30% каналов, День 2: 30%, День 3: остальные
- **Между подписками — активность** — после подписки на канал прочитать последние 2-3 поста (send_read_acknowledge). Реальный человек подписывается и сразу листает
- **Отписки** — иногда случайно отписаться от одного канала и подписаться заново через час. Реальные люди так делают
- **Не подписываться ночью** — подписки только в активные часы (8:00-23:00)

### 7.4 Реакции — требования к человечности

- **Не ставить реакцию на старые посты** — только на посты не старше 24 часов
- **Разные эмодзи** — не только 👍, вес эмодзи должен зависеть от контента (позитивный пост → ❤️🔥, новость → 😮🤔)
- **Не реактить каждый пост** — пропускать 30-50% постов
- **Паузы** — между реакциями от одного аккаунта минимум 2-5 минут
